C++ Lambda 与 Netlink Socket 学习笔记

📅 2026-06-25 · 🏷️ C++ / Linux / 网络编程

概述

本文记录了一个网络端口状态监控模块的实现。该模块通过 Linux Netlink Socket 监听内核网络接口的 up/down 变化，检测到变化后通过 Lambda 回调构建 JSON 消息，最终通过 MQTT 协议上报到云端。

  • C++ Lambda 表达式 — 作为回调函数实现模块间解耦
  • Netlink Socket — Linux 内核与用户空间的通信机制
  • 事件循环（EventLoop） — 单线程异步 I/O 模型
  • 状态机设计 — 快照初始状态 + 增量变化检测

1. C++ Lambda 表达式作为回调

什么是 Lambda

Lambda 表达式是 C++11 引入的特性，本质上就是创建一个匿名函数并可以立即使用：

[捕获列表](参数列表) {
    // 函数体
}

实际用法

Lambda 被用来作为构造函数的回调参数：

_port_monitor = new MqttPortMonitor(
      [this](const MqttPortMonitor::PortEvent& ev) {
          on_port_event(ev);
      });

💡 逐行理解

  • [this] — 捕获 this 指针，Lambda 内可调用当前类成员函数
  • (const PortEvent& ev) — 接收端口事件的常量引用
  • { on_port_event(ev); } — 调用当前类的处理方法

Lambda 的好处：就地定义、就地使用，不用单独写函数再传进去。

类原型

class MqttPortMonitor {
public:
    struct PortEvent {
        std::string port_id;      // "ge-1/1/1"
        std::string status;       // "up" or "down"
        std::string prev_status;  // "up" or "down"
    };
    typedef std::function<void(const PortEvent&)> EventCallback;
};

MqttPortMonitor::MqttPortMonitor(EventCallback cb)
    : _callback(cb), _nl_fd(-1) {}

2. Netlink Socket — 监听内核端口状态

创建并绑定

bool MqttPortMonitor::start() {
    _nl_fd = socket(AF_NETLINK, SOCK_DGRAM, NETLINK_ROUTE);
    if (_nl_fd < 0) return false;

    struct sockaddr_nl addr;
    memset(&addr, 0, sizeof(addr));
    addr.nl_family = AF_NETLINK;
    addr.nl_groups = RTMGRP_LINK;  // 订阅链路变化

    if (bind(_nl_fd, (struct sockaddr*)&addr, sizeof(addr)) < 0) {
        close(_nl_fd); _nl_fd = -1;
        return false;
    }
    // ... 快照初始状态
}

📌 关键参数

参数含义
AF_NETLINK内核与用户空间通信的协议族
NETLINK_ROUTE路由子系统，获取网络接口变化
RTMGRP_LINK多播组：链路层 up/down 事件

快照当前端口状态

// 遍历 /sys/class/net/ 记录初始状态
DIR* dir = opendir("/sys/class/net");
while ((ent = readdir(dir)) != NULL) {
    // 读取 /sys/class/net/{name}/operstate
    _link_state[kname] = (strcmp(state, "up") == 0);
}
closedir(dir);

🎯 为什么要快照？

Netlink 只推送后续变化。bind 之前已存在的状态不会推送。所以先遍历记下初始值，后续才能对比出 prev_status。

3. 事件循环集成

int fd = _port_monitor->get_fd();
if (fd >= 0) {
    _eventloop.add_ioevent_cb(fd, IOT_READ,
        callback(this, &Target::on_port_monitor_readable));
}

把 Netlink fd 注册到事件循环的 IOT_READ。内核推送变化时 fd 可读，自动触发回调。没有另起线程，单线程无锁。

4. on_readable() — 解析 Netlink 消息

void MqttPortMonitor::on_readable() {
    char buf[4096];
    ssize_t len = recv(_nl_fd, buf, sizeof(buf), MSG_DONTWAIT);
    if (len <= 0) return;

    for (struct nlmsghdr* nh = ...; NLMSG_OK(nh, len); nh = NLMSG_NEXT(nh, len)) {
        if (nh->nlmsg_type != RTM_NEWLINK) continue;

        // 解析 ifinfomsg，提取 flags 和接口名
        struct ifinfomsg* ifi = (struct ifinfomsg*)NLMSG_DATA(nh);
        std::string ifname = /* 从 rtattr 中提取 IFLA_IFNAME */;

        bool now_up = is_link_up(ifi->ifi_flags);

        // 对比 _link_state，状态没变则跳过
        auto it = _link_state.find(ifname);
        if (it != _link_state.end() && it->second == now_up)
            continue;

        // 状态变了，触发回调
        PortEvent ev{logical, now_up?"up":"down", prev_up?"up":"down"};
        if (_callback) _callback(ev);
    }
}

is_link_up

bool is_link_up(unsigned int flags) {
    return (flags & IFF_LOWER_UP) != 0;  // 物理层有载波
}

🎯 IFF_LOWER_UP vs IFF_UP

IFF_UP = 管理员启用了接口（网线可能没插）。IFF_LOWER_UP = 物理层检测到载波信号，网线真正连通了。

7. 设计巧妙之处

✨ 值得学习的设计

  • 事件驱动 + 零轮询 — Netlink 内核推送，CPU 占用近零
  • Lambda 回调解耦 — Monitor 不知道上层是 MQTT/HTTP/日志，零耦合
  • 快照 + 增量 — start() 记录初始状态，只在真正变化时触发
  • 单线程事件循环 — fd 注册到 EventLoop，无锁无线程竞争
  • IFF_LOWER_UP — 物理层载波判断，比 IFF_UP 更准确
  • 防御性编程 — 首次发现未知接口视为变化；MQTT 未连接时优雅丢弃
